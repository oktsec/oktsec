## Executive Summary

Oktsec is a security proxy and MCP gateway for AI agent-to-agent communication. Single Go binary, no LLM, deterministic. 9-stage security pipeline, 4 operational modes, full web dashboard, 175 detection rules.

**Codebase:** 28,230 LOC Go | 396 tests | 18 packages | all tests green

## What's Built (complete inventory)

### Security Pipeline (handler.go)
1. Rate limiting — per-agent sliding window
2. Identity verification — Ed25519 signatures
3. Agent suspension check
4. ACL evaluation — YAML-based, default-deny
5. Content scanning — 175 rules via Aguara engine
6. Blocked content filter — per-agent category enforcement
7. Verdict escalation — history-based (3+ blocks in 1h)
8. Audit log write — SQLite batched (~90K inserts/sec)
9. Anomaly detection — background risk scoring

### Operational Modes
- `oktsec serve` — HTTP proxy server
- `oktsec proxy` — stdio proxy wrapping MCP servers
- `oktsec gateway` — Streamable HTTP MCP gateway fronting backends
- `oktsec mcp` — MCP tool server (6 tools)

### Dashboard (HTMX, self-hosted, offline-capable)
- Overview with detection rate, unsigned %, latency, top rules
- Events with agent/time-range filters and export
- Agents with risk scores, traffic stats, communication partners
- Rules with full-page detail, inline testing, per-category webhooks
- Enforcement with create/edit/delete overrides
- Settings with tabbed config editing
- Audit with health score, severity breakdown, remediation
- Mobile-responsive (3 breakpoints), auth via access codes, CSRF

### Detection Rules (175 total)
- 148 Aguara rules (prompt injection, credential leaks, PII, exfiltration, code exec, supply chain)
- 6 inter-agent rules (IAP- prefix)
- 15 OpenClaw rules (OCLAW-001 to OCLAW-015)
- 6 NanoClaw rules

### Infrastructure
- Ed25519 keypair management, PEM format, replay protection (5-min window)
- SSRF hardening (21 CIDR blocks, DNS rebinding protection)
- Credential redaction (15 patterns)
- Symlink rejection, size-limited reads
- 17 MCP client auto-discovery
- Go SDK for programmatic access
- curl|bash installer, cross-platform binaries
- Deployment auditor (41 checks, A-F grading, SARIF output)
- Traffic generator for demos

### Demos
- `demo/traffic-gen` — realistic agent traffic simulation
- `demo/agentcard-gateway` — Agent Card + gateway integration
- `demo/sandbox-nanoclaw` — Docker containerized NanoClaw setup

## The Core Problem: Bridging Build to Adoption

Everything above works. The gap is: **how does someone go from "I have agents" to "my agents are secured by Oktsec"?** Today that requires reading source code, understanding the config schema, and manually wiring things together. The product is technically complete but operationally opaque.

---

## Real-World Use Cases (and what's missing for each)

### Use Case 1: "I run Claude Desktop / Cursor with MCP servers"

**What they want:** Scan every tool call my MCP servers handle. Alert me if something sketchy passes through.

**What exists:** `oktsec proxy` wraps any stdio MCP server. `oktsec discover` finds installed MCP configs.

**What's missing:** ~~All resolved in v0.8.0.~~
- ~~**One-command setup.**~~ DONE: `oktsec setup` discovers, generates config, wraps all clients in one step.
- ~~**Zero-config defaults.**~~ DONE: `setup` generates `oktsec.yaml` with observe mode, all rules enabled, no identity required.
- ~~**Post-install feedback.**~~ DONE: `setup` prints detailed summary of what's protected and next steps.
- **Webhook to desktop notification.** Not yet — user must configure Slack/Discord webhook manually.

### Use Case 2: "I'm building a multi-agent system"

**What they want:** Agents authenticate to each other, messages get scanned, bad actors get blocked.

**What exists:** Full pipeline — identity, ACL, scanning, escalation, audit.

**What's missing:**
- **Integration guide per framework.** Step-by-step for: CrewAI, AutoGen, LangGraph, raw OpenAI function calling. Show exactly where to add the HTTP call or SDK import.
- **SDK in more languages.** Go SDK exists. Python and TypeScript SDKs would cover 90% of agent builders.
- **Agent onboarding flow.** `oktsec add-agent --name "researcher" --generate-keys` should create keys, update config, and print the signing snippet for that agent's language.

### Use Case 3: "I run an MCP gateway for my team"

**What they want:** Centralized gateway where all agents connect, with per-agent permissions and audit trail.

**What exists:** `oktsec gateway` with tool discovery, namespacing, allowlists, response scanning.

**What's missing:**
- **Team management.** Today: edit YAML to add agents. Need: dashboard UI for agent CRUD, key distribution, allowlist management.
- **Onboarding handshake.** Agent connects to gateway, presents credentials, gets approved (or queued for approval). Today: pre-configured trust only.
- **Billing/quota hooks.** For commercial deployments: per-agent usage tracking, rate limit tiers.

### Use Case 4: "I need compliance/audit for my AI deployment"

**What they want:** Prove to security team / auditors that agent communication is monitored and controlled.

**What exists:** `oktsec audit` (41 checks, A-F grade, SARIF), full audit trail, quarantine queue.

**What's missing:**
- **Compliance report export.** One-click PDF/HTML report: "Here's what we scan, here's our policy, here are last 30 days of stats." For SOC2, ISO 27001 conversations.
- **Retention policies.** Auto-archive/delete audit logs older than N days. Today: grows forever.
- **SIEM integration.** Syslog or OTLP export for Splunk/Datadog/ELK. Today: SQLite only.

## Adoption Roadmap: What to Build Next

### Phase 1: Frictionless First Experience — DONE (v0.8.0)

1. **`oktsec setup`** — DONE. One command: discover + init + wrap all + summary with next steps.
2. **`oktsec wrap --all`** — DONE. Wraps all discovered clients at once (13 wrappable clients).
3. **Post-wrap welcome** — DONE. Detailed terminal output: what's protected, mode, agent count, next steps.
4. **Quick-start guide** — DONE. `DOCS/quick-start.md` — "Secure your MCP servers in 2 minutes."
5. **Shared audit trail** — DONE. Proxy and serve share `~/.oktsec/oktsec.db` so dashboard shows proxy activity.
6. **Config passthrough** — DONE. Wrapped proxies receive `--config` flag, enabling allowed_tools and all config features.

### Phase 2: Developer Experience (weeks 3-4)

5. **Python SDK** — `pip install oktsec`. Sign messages, send through proxy, query audit. Most agent frameworks are Python.
6. **TypeScript SDK** — `npm install oktsec`. Same API surface. Covers Node.js agent builders.
7. **Framework integration examples** — Working code for CrewAI, AutoGen, LangGraph, Mastra showing oktsec wired in.
8. **`oktsec add-agent`** — CLI command to register an agent: generate keys, update config, print code snippet.

### Phase 3: Operations & Observability (weeks 5-6)

9. **Prometheus `/metrics`** — Counters: verdicts by type, latency histogram, active agents, rule hits. Grafana dashboard template included.
10. **Webhook presets** — Built-in templates for Slack, Discord, PagerDuty, email. `oktsec init` offers to configure.
11. **Syslog/OTLP export** — Stream audit events to external systems.
12. **Retention policies** — `retention_days: 90` in config. Auto-prune old events.

### Phase 4: Team & Scale (weeks 7-8)

13. **Dashboard agent management** — CRUD agents from UI. Generate keys, set allowlists, view per-agent stats.
14. **OIDC/OAuth for dashboard** — Replace access codes with real auth for team deployments.
15. **Config hot-reload** — Watch `oktsec.yaml` for changes, apply without restart.
16. **Compliance report** — Export dashboard stats + policy config as PDF/HTML for auditors.

---

## Key Insight

The product is feature-complete for the security pipeline. The bottleneck is **the last mile**: making it trivial to install, configure, and integrate. Every feature we've built is useless if the setup friction prevents people from reaching it.

Priority order: **init wizard > quick-start docs > Python SDK > Prometheus metrics > everything else.**
